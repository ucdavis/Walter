# Accrual Notifications Implementation Plan

## Goal

Build v1 accrual notifications as a scheduled background workflow that:

- Sends monthly employee emails to people whose latest vacation accrual status is **Approaching Cap** or **At Cap**.
- Sends monthly accrual overview emails to active users explicitly assigned the **AccrualViewer** role.
- Persists outbound delivery state in the app database behind `AppDbContext`.
- Uses durable dedupe keys so reruns are idempotent.

## Non-Goals

- No user notification preferences or unsubscribe model.
- No admin UI for queue inspection or repair.
- No Slack or non-email channels beyond a schema shape that can tolerate later channels.
- No department-scoped viewer reports in v1.
- No direct coordination with the ADF accrual refresh pipeline.
- No rendered subject/body persistence after send.

## Data Model

Add an `OutboundMessage` entity/table to `server.core`.

Suggested columns:

- `Id`: primary key.
- `RunId`: GUID generated by the monthly queueing job for correlation.
- `NotificationType`: concrete workflow, such as `accrual.employee` or `accrual.viewer-report`.
- `RecipientType`: coarse recipient class, such as `Employee` or `AccrualViewer`.
- `Channel`: required, default `Email`.
- `RecipientEmail`: required delivery address.
- `RecipientName`: optional display name.
- `Status`: `Pending`, `Processing`, `Sent`, `Retry`, or `DeadLetter`.
- `DedupeKey`: immutable unique key for the logical notification.
- `TemplateKey`: rendering key, such as `accrual.employee.staff.v1`.
- `TemplateVersion`: integer, starts at `1`.
- `PayloadVersion`: integer, starts at `1`.
- `PayloadJson`: structured renderable payload, retained indefinitely and allowed to contain PII.
- `NotBeforeUtc`: earliest send/retry time.
- `LockedUntilUtc`: processing lease expiry.
- `LockId`: GUID set during claim and required for completion updates.
- `AttemptCount`: incremented for each processing/send attempt.
- `LastError`: sanitized, truncated error summary.
- `ProviderMessageId`: SMTP/SparkPost provider message id when available.
- `CreatedUtc`: UTC creation time.
- `SentUtc`: UTC success time.

Indexes:

- Unique index on `DedupeKey`.
- Query index for sender claim, likely `(Status, NotBeforeUtc, LockedUntilUtc)`.
- Optional operational indexes on `RunId`, `NotificationType`, `RecipientType`, and `CreatedUtc`.

Retention:

- Keep rows forever in v1.
- Treat `PayloadJson` as sensitive application data.

## Accrual Data Changes

Extend the accrual report/datamart shape so `EmployeeAccrualBalanceRecord` includes employee email.

Rules:

- The accrual snapshot is the source for employee notification email.
- For employees with multiple rows, use the most recent non-empty email.
- Do not resolve employee email through IAM or Graph during queue generation.
- Dedupe employee notifications by employee ID and snapshot date, not email.

Example employee dedupe key:

```text
accrual:employee:{employeeId}:{snapshotAsOfDate:yyyy-MM-dd}
```

## Employee Eligibility

Use the existing accrual report semantics:

- **Approaching Cap**: at least 80% of cap and below at-cap threshold.
- **At Cap**: at least 96% of cap.
- Notify every monthly run while the employee remains **Approaching Cap** or **At Cap**.
- Send at most one employee notification per employee per accrual snapshot.
- If the latest accrual snapshot was already notified, the monthly generator no-ops and logs that there is no new snapshot.

One employee should receive one email per snapshot even if they have multiple appointments or positions.

## Employee Groups

Derive employee group from the same normalized classification used by accrual reporting.

- `FacultyAcademic`: `FY Acad Admin`, `FY Acad Coord`, `FY Faculty`, `FY Researcher`.
- `Staff`: `PSS`, `MSP`, `SMG`, and staff-like classifications.
- `Generic`: fallback for unmapped classifications.

Employee group chooses body wording and leave-request CTA. Unknown classifications should use `Generic`, not be skipped.

## Viewer Reports

Generate viewer report messages in the same monthly queueing run as employee notifications.

Recipients:

- Active app users explicitly assigned `AccrualViewer`.
- Do not include `Admin` users by authorization bypass unless they also explicitly have `AccrualViewer`.

Content:

- Reuse the existing accrual overview calculation so email and UI agree.
- Include snapshot date, approaching count, at-cap count, total employees/departments, lost cost summary, waste rate, and department breakdown.
- Link back to Walter using configured `WalterBaseUrl`.
- Do not include employee notification delivery status.

Example viewer report dedupe key:

```text
accrual:viewer-report:{userId}:{snapshotAsOfDate:yyyy-MM-dd}
```

## Queue Generation

Create a separate worker or Azure Functions project in the solution.

Monthly queueing function:

1. Load the latest accrual data needed for the same reporting window used by the UI.
2. Build the accrual overview and employee snapshots.
3. Determine the latest accrual snapshot date.
4. Compute all employee and viewer recipients in memory.
5. Validate recipient emails.
6. Bulk enqueue valid `OutboundMessages` through a queueing service.
7. Treat existing dedupe keys as idempotent skips, not errors.
8. Log counts for queued, skipped invalid email, skipped duplicate, and other skip reasons.

Failure behavior:

- A single invalid recipient must not fail the full generation run.
- Employee recipient skips must not block viewer reports.
- If shared accrual snapshot calculation fails, fail generation for both employee and viewer messages.

Dry run:

- Support a generator dry-run mode that computes recipients and logs counts/skips without inserting messages.

## Sender

Create a recurring sender function that runs every configured interval, initially 15 minutes.

Claiming:

- Use raw SQL against the app DB for atomic multi-worker claim.
- Encapsulate claim/update logic behind an `IOutboundMessageQueue` service.
- Claim configurable batch size, default `500`.
- Eligible rows are:
  - `Pending` or `Retry` where `NotBeforeUtc <= now`.
  - `Processing` where `LockedUntilUtc < now`.
- Claim by setting `Status = Processing`, assigning a new `LockId`, and extending `LockedUntilUtc`.
- Use SQL Server locking hints such as `UPDLOCK`, `READPAST`, and `ROWLOCK`.

Completion:

- Mark rows sent/retry/dead by matching both `Id` and `LockId`.
- This prevents an old worker from overwriting a row after its lease has expired and another worker reclaimed it.

Concurrency:

- Sender batch size is separate from SMTP send concurrency.
- Make send concurrency configurable, default around `10`.

Disabled delivery:

- Add `SenderEnabled=false` as an emergency/launch switch.
- When disabled, sender should not claim rows.
- Dev/test should normally use Mailtrap SMTP settings rather than rewriting recipient data in the database.

## Delivery And Retry

Use SMTP/SparkPost through a delivery abstraction.

Retry rules:

- `Pending`, `Processing`, `Sent`, `Retry`, `DeadLetter` are the only v1 statuses.
- Max attempts: `5`.
- Transient failures increment `AttemptCount`, set `Status = Retry`, and set `NotBeforeUtc` using exponential backoff with jitter.
- Permanent recipient/template/provider validation failures go directly to `DeadLetter`.
- Unknown `NotificationType` or unsupported `TemplateKey` is permanent and should become `DeadLetter`.
- `AttemptCount` increments for each actual processing/send attempt, including permanent failures.
- Reclaiming stale `Processing` rows does not increment `AttemptCount`.
- Store sanitized/truncated `LastError`; full details go to logs/telemetry.

No v1 repair path:

- There is no admin page or system endpoint to requeue `DeadLetter` rows.
- Manual DB intervention or a later operational feature would be required.

## Templates And Configuration

Templates are versioned in code for v1.

Store structured payload plus `TemplateKey` and versions. Render at send time.

Do not store rendered subject/body after send.

Configuration should include:

- Employee email from address.
- Reply-to or support contact.
- Policy citation text, defaulting to `PPSM 2.210`.
- Faculty academic CTA label and URL.
- Staff CTA label and URL.
- Generic CTA label and URL.
- Walter base URL for viewer report links.
- SMTP host, port, credentials, TLS settings, and provider options.
- Sender enabled switch.
- Generator dry-run switch.
- Sender batch size.
- Sender concurrency.
- Retry/backoff settings.

Employee email guidance:

- Visible copy should come from CAES HR or configured sender identity.
- Include “automated notification” language.
- Employee emails should link to leave-request systems, not Walter.
- Do not include lost-cost dollars in employee emails.
- Subject should vary by status, not employee group.

Suggested subject patterns:

```text
Action Needed: Your Vacation Accrual Is Approaching the Maximum
Action Needed: Your Vacation Accrual Is At or Above the Maximum
```

Viewer report guidance:

- Link back to Walter accrual reporting.
- Keep it focused on system accrual risk and department breakdown.

## Non-Production Safety

- In non-production, require either generator dry-run, Mailtrap/test SMTP settings, or a consciously enabled sender configuration.
- Do not silently rewrite `RecipientEmail` in the database for non-production.
- Preserve original recipient metadata in queued rows even when SMTP delivery goes to Mailtrap.
